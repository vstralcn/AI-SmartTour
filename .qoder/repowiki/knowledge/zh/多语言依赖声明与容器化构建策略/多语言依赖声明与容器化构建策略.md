本仓库采用多语言、多模块的依赖管理方式，各子项目独立声明依赖并通过 Docker Compose 统一编排：

- **Python 后端（backend）**：使用 `pyproject.toml` 作为唯一依赖清单，基于 setuptools 构建系统。核心运行时依赖通过 `dependencies` 列表精确锁定版本（如 `aiosqlite==0.21.0`、`pypdf==5.7.0`、`python-docx==1.2.0`），可选功能通过 `[project.optional-dependencies]` 拆分：`rag` 组包含 langchain/chromadb/sentence-transformers 等向量检索组件，`dev` 组包含 pytest/ruff 开发工具。未生成 `requirements.txt`，但 `.dockerignore` 排除了 `*.egg-info`、`.venv`、`__pycache__` 等缓存目录。
- **数字人服务（digital_human）**：使用传统 `requirements.txt`，仅声明 FastAPI + uvicorn + python-multipart 三个最小依赖，保持轻量。
- **前端应用（admin-panel / tourist-app）**：各自维护独立的 `package.json`，分别声明 Vue3 + Element Plus（管理端）和 Vue3 + Three.js/VRM（游客端）生态；均附带 `package-lock.json`（lockfileVersion=3）以固定完整依赖树，确保构建可重现。
- **容器化约束**：`docker-compose.yml` 中所有服务通过 `build.context` 从源码构建镜像，不直接拉取预编译镜像，因此依赖安装发生在每个服务的 Dockerfile 内（由各自包管理器执行）。环境变量通过 `${VAR:-default}` 注入外部配置（数据库、Redis、LLM API Key 等）。

**约定与约束**：
- Python 侧禁止在代码中硬编码版本号以外的来源，新增依赖需同步更新 `pyproject.toml` 对应分组；RAG 相关库必须放入 `rag` optional group，避免默认安装体积膨胀。
- 前端侧升级依赖后必须提交对应的 `package-lock.json`，禁止只改 `package.json` 不锁版本。
- 数字人服务因 GPU 依赖较重，通过 docker-compose `profiles: [gpu]` 按需启用，非 GPU 环境默认不拉起该服务。
- 无私有 PyPI/NPM 源或 vendoring 策略，全部依赖来自官方公共注册表。
