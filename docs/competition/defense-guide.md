# SmartTour Copilot 比赛答辩手册

## 1. 答辩总原则

回答建议遵循“**结论—证据—边界**”三段式：

1. 先用一句话正面回答。
2. 再用系统实现或量化指标证明。
3. 最后主动说明当前边界和下一步，不夸大完成度。

避免只回答“我们用了 Qwen、RAG、数字人”。评委关心的是技术如何解决问题、是否真实实现、效果如何验证。

## 2. 高频评委问题与推荐回答

### Q1：你们和普通 ChatGPT 导游有什么区别？

**推荐回答：**

普通 ChatGPT 主要完成内容生成，而我们的系统完成的是受约束的景区服务任务。它先读取游客的时长、兴趣、同行人和行动能力，再由 Agent 调用景区知识、路线、画像和反馈工具；事实回答必须附景区知识来源，证据不足时拒答。对话结束后，匿名服务事件还会进入后台，形成知识缺口和服务质量指标。因此我们的核心不是“换一个导游提示词”，而是“可信知识 + 可执行工具 + 运营闭环”。

**可展示证据：** Agent 工具调用时间线、带来源回答、后台未命中问题。

### Q2：为什么需要数字人？语音助手不够吗？

**推荐回答：**

数字人的价值不只是视觉效果，而是提供统一、可感知的服务角色。景区讲解涉及文化传播、儿童研学、老人服务和投诉安抚，不同场景需要不同语速、措辞和情绪表达。我们的游客状态会驱动数字人的角色、语音风格和表情；同时它可以承载景区 IP，形成品牌识别。为了控制成本，数字人服务异常时会降级为静态形象加语音，不影响核心问答。

**可展示证据：** 同一问题在儿童、老人、投诉三种状态下的输出对比。

### Q3：RAG 如何保证准确率？

**推荐回答：**

我们不只做单路向量检索，而是分四步保证：第一，高频票务和开放时间使用精确 FAQ 或结构化数据；第二，BGE-M3 混合召回景区文档；第三，Reranker 精排；第四，按相关性阈值决定回答还是拒答，并展示来源。评测使用 200 道人工标注问题和标准证据，分别计算事实准确率、Recall@5、忠实度和正确拒答率。这样可以定位问题究竟出在检索还是生成，而不是只报告一个模糊准确率。

**可展示证据：** 离线评测报告、来源卡片、知识库外问题拒答案例。

### Q4：你们所说的 Agent 具体做了什么？

**推荐回答：**

Agent 负责把自然语言目标转换为受控任务流程。例如“带老人玩 3 小时，少爬坡”会被拆成画像读取、景点知识检索、路线规划和约束校验。每一步只能调用我们定义的白名单工具，并保存输入、结果、耗时和状态。Agent 不直接执行任意代码，也不直接修改数据库。我们选择 LangGraph 单编排器，而不是复杂多 Agent，是为了在比赛周期内保证可观察、可测试和稳定。

**可展示证据：** 结构化状态、工具参数、执行轨迹和失败降级。

### Q5：路线推荐为什么算个性化？是不是简单标签排序？

**推荐回答：**

兴趣只是一项软偏好，真正决定路线可用性的是硬约束。系统先检查时长、同行人、行动能力、景点开放状态和路径顺序，再在可行路线中优化兴趣匹配和体验。每个景点都显示推荐理由。游客在途中说“累了”或“只剩 1 小时”时，只重规划未完成部分。因此它不是静态标签排序，而是可解释的约束规划。

**可展示证据：** 同一兴趣在不同时间和行动能力下的路线对比。

### Q6：你们的情绪识别靠谱吗？

**推荐回答：**

我们没有把情绪识别包装成心理诊断。它只用于服务交互策略，识别开心、关切、讲解等低风险状态。实现上采用规则保底和模型结构化分类，并允许用户行为覆盖系统判断。我们评测的是状态分类正确率和用户对表达自然度的评分，而不是声称识别复杂心理状态。

**可展示证据：** 情绪状态机和几类标准测试语句。

### Q7：当前哪些数据是真实的，哪些是 Demo？

**推荐回答：**

当前游客端、管理端、文本流式问答、基础检索和兴趣路线已经具备原型；ASR、TTS、MuseTalk 与部分运营指标仍在接入或使用演示数据。我们在 README 中逐项标注了“原型可用、基础可用、演示数据、接口预留”，不会把占位接口当作完成成果。30 天计划优先将核心演示链路和后台关键指标替换为真实数据。

**可展示证据：** README 状态矩阵和迭代验收清单。

### Q8：为什么选 Qwen、BGE-M3 和 MuseTalk？

**推荐回答：**

选择标准是中文效果、开源可控、部署成本和现有生态。Qwen 适合中文任务和结构化工具调用；BGE-M3 能统一支持 Dense、Sparse 和多语言检索；MuseTalk 可以基于 2D 形象完成实时口型，成本低于 3D 数字人。我们没有追求模型数量，而是为每个模块选择一个能在学生项目周期内落地的方案，并设计替代和降级路径。

### Q9：为什么不用更复杂的多 Agent？

**推荐回答：**

多 Agent 只有在角色边界明确、并行收益大于通信成本时才有价值。景区导览的核心任务可以由单编排器和受控工具完成。过早引入多 Agent 会增加延迟、Token 消耗和故障点，却不一定提升游客体验。因此当前采用单 Agent 状态机，后续只有在跨景区协作、复杂行程编排等场景明确后才拆分专业 Agent。

### Q10：项目如何商业落地？

**推荐回答：**

客户是景区、博物馆和文旅综合体，交付分为 SaaS 年订阅和私有化部署。景区购买的不只是数字人形象，而是统一知识口径、覆盖长尾服务时段和获得游客语义洞察。最小商业闭环是游客扫码使用、AI 完成高频咨询、后台发现知识缺口、运营人员更新内容、服务指标提升。增值项包括景区 IP 形象、专属音色、知识治理和月度运营报告。

**可展示证据：** 单景区部署图、后台闭环、成本与服务量估算。

### Q11：如何证明能为景区降本增效？

**推荐回答：**

比赛阶段不直接承诺未经试点验证的降本比例，而是先测量可转化指标：AI 独立完成咨询占比、人工接管率、重复问题覆盖率、知识更新时间和单次服务成本。试点后再用上线前后的人工咨询量和满意度对比计算实际收益。这样的指标更可核验，也便于景区做采购决策。

### Q12：遇到模型幻觉怎么办？

**推荐回答：**

我们从数据、检索、生成和服务四层控制。结构化事实由工具返回；文档回答必须引用证据；相关性不足时拒答；高风险或投诉场景可转人工。后台记录低置信度和失败会话，补充知识后自动回归。核心目标不是让模型“任何问题都回答”，而是让系统知道什么时候不能回答。

### Q13：景区知识更新后多久生效？

**推荐回答：**

目标是在 60 秒内生效。文档上传后完成解析、切片、向量化和版本登记，新版本通过小规模回归检查后切换为可用。旧版本保留以便回滚。票价和开放时间等高频结构化信息可以即时更新，不必等待整份文档重新处理。

### Q14：如何保护游客隐私？

**推荐回答：**

系统默认使用匿名会话 ID，只采集完成导览所需的兴趣、时长、同行结构和行动偏好，不要求真实姓名。游客可查看和重置画像，运营分析只展示聚合结果。部署时设置数据保留期限、分级权限和日志脱敏；有更高合规要求的景区可选择私有化部署。

### Q15：网络或 GPU 服务故障时还能用吗？

**推荐回答：**

系统按能力分级降级。数字人失败时切换到静态形象加 TTS；TTS 失败时保留流式文字；向量服务失败时使用缓存 FAQ；LLM 不可用时仍能查询结构化票务、开放时间和设施信息。核心服务不依赖单个高成本模型，因此现场演示和实际部署更稳定。

### Q16：你们的技术创新是否只是集成开源模型？

**推荐回答：**

模型集成本身不是主要创新。我们的创新在系统方法：把游客状态、可信知识、约束规划、数字人表达和运营反馈连接为可评测闭环。评测会分别验证检索、工具选择、任务完成、延迟和满意度。比赛作品的价值不是重新训练所有底层模型，而是用合理架构解决行业问题并证明效果。

### Q17：如果只有 30 天，最重要的功能是什么？

**推荐回答：**

优先完成四件事：第一，建立具体景区知识与评测集；第二，完成可信 RAG 和来源拒答；第三，实现单 Agent 调用路线、画像和反馈工具；第四，接通一个稳定的语音数字人链路并把真实事件送入后台。暂缓复杂 3D、多 Agent 和重型预测平台。这样能同时覆盖创新、完整、实用和商业价值。

## 3. 5 分钟答辩演示脚本

### 0:00–0:40 问题与定位

“传统导览只能给固定内容，通用大模型又存在知识过时和幻觉。智游灵境让 AI 不只回答，而是基于景区可信知识完成导览任务，并把服务结果反馈给景区运营。”

### 0:40–2:10 游客复杂任务

输入：

> 我带父母来玩，只有 3 小时，喜欢古建筑，尽量少爬坡。

展示：

1. Agent 提取画像和约束。
2. 调用知识与路线工具。
3. 返回路线、总时长、推荐理由和来源。
4. 数字人以较慢语速播报首段。

### 2:10–2:50 动态重规划

输入：

> 现在下雨了，而且我们只剩 1 小时。

展示未完成路线被重新规划，并解释删除景点的原因。

### 2:50–3:35 可信 RAG

1. 提问知识库内问题，展示来源。
2. 提问一个知识库外问题，展示拒答。
3. 后台出现一条未命中问题。

### 3:35–4:25 运营闭环

1. 管理员补充一条新知识。
2. 执行单题测试。
3. 游客再次提问并正确命中。
4. 展示服务质量中的命中、延迟和反馈。

### 4:25–5:00 商业与总结

“这套系统通过 SaaS 或私有化方式交付。景区获得统一知识、长尾服务和真实运营洞察；游客获得个性化、可信、自然的导览。我们的核心不是数字人外壳，而是可执行、可评测、可运营的 AI 服务闭环。”

## 4. 现场演示准备清单

- 固定一份可核验景区知识快照。
- 准备 3 条主问题和 2 条备用问题。
- 预热模型、向量库和数字人服务。
- 健康检查展示后立即隐藏，不占用演示时间。
- 准备静态形象 + TTS 降级。
- 准备本地录屏，网络故障时作为最后保障。
- 清空旧会话，确保后台增量数据明显。
- 所有指标截图标明版本、日期、样本数。
- 连续执行 20 次标准流程并记录成功率。
- 演示者只讲“已经实现并可展示”的能力。

## 5. 答辩禁区

1. 不说“完全杜绝幻觉”，应说“通过引用、拒答和人工接管降低风险”。
2. 不说“替代讲解员”，应说“覆盖重复咨询和长尾时段，复杂服务转人工”。
3. 不报没有测试口径的“准确率 99%”。
4. 不把 Mock 数据、占位接口和规划功能描述为已完成。
5. 不用大量模型名替代业务价值和技术解释。
6. 不回避 GPU 成本、隐私和网络故障，应主动展示降级和部署方案。
