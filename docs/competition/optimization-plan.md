# SmartTour Copilot 比赛竞争力与产品化优化方案

## 1. 评委视角总体诊断

### 1.1 当前优势

1. 赛题契合度高：具备游客端、管理后台、问答、路线、数字人和分析模块的完整产品外形。
2. 技术路线合理：Vue 3、FastAPI、Qwen、RAG、BGE-M3、MuseTalk 均有成熟生态，适合学生团队。
3. 演示链路清晰：语音或文本提问、数字人回答、路线推荐、后台分析天然适合现场展示。
4. 具备 B 端价值：知识库管理和运营后台使项目不局限于游客端 Demo。

### 1.2 当前主要不足

| 不足 | 代码现状 | 评委可能产生的判断 | 优化原则 |
|---|---|---|---|
| 定位偏“功能集合” | README 主要罗列问答、数字人、推荐 | 与常见大模型套壳项目差异不足 | 强调“游客服务执行 + 景区运营闭环” |
| Agent 能力缺失 | 当前为固定的 RAG 后对话链 | 所谓智能只停留在生成文本 | 增加轻量任务编排和白名单工具调用 |
| RAG 名称大于实现 | 当前主要是 FAQ/关键词检索 | ≥90% 准确率缺少证据 | 落地向量召回、重排、引用、拒答和评测集 |
| 数字人链路未接通 | ASR/TTS/MuseTalk 均为接口占位 | 数字人可能被认为只是头像动画 | 先完成最短可用链路，并提供降级方案 |
| 运营数据为 Mock | 后台图表使用演示数据 | 商业价值缺少真实数据闭环 | 从对话事件生成最小真实指标 |
| 个性化较弱 | 仅保存兴趣标签 | 路线仍像规则排序 | 增加时长、同行人、行动能力和反馈更新 |
| 缺少工程证据 | 无固定评测集和压测报告 | 指标容易被视为宣传口号 | 建立可重复执行的离线与链路评测 |
| 场景数据泛化 | 默认知识为“某景区”虚构数据 | 难以证明试点价值 | 选一个具体景区构建可核验演示知识集 |

### 1.3 竞争力判断

当前项目具备“完整 AI Demo”的基础，但要进入高水平比赛作品，需要跨越三个台阶：

```text
功能演示
  → 有证据的技术系统：工具调用、引用、评测、降级
  → 可运营的行业产品：知识治理、质检、数据闭环、部署与商业模式
```

决定名次的不是继续堆叠模型，而是把现有模块连接成可验证的业务闭环。

## 2. 项目定位升级

### 2.1 推荐名称

**中文名：智游灵境**  
**英文名：SmartTour Copilot**

### 2.2 推荐副标题

**面向智慧景区的多模态智能体数字人服务与运营平台**

“服务与运营平台”比“景区导览数字人”更能体现 B 端价值；“智能体”说明系统具备任务执行能力；“多模态数字人”保留赛题识别度。

### 2.3 推荐项目简介

> 智游灵境是一套面向景区游客与运营管理方的多模态 AI 服务平台。系统以景区知识库为可信信息底座，通过导览 Agent 理解游客目标、规划任务并调用知识检索、路线规划、游客画像和服务反馈等工具，再由具备语音、口型和情绪表达能力的数字人完成自然交互。同时，系统将匿名服务事件沉淀为知识缺口、游客偏好和服务质量指标，帮助景区持续优化内容与运营。项目支持云端与私有化部署，适用于景区、博物馆和文旅综合体。

## 3. 高价值创新点

### 3.1 创新一：可解释的导览任务 Agent

**技术实现**

- 使用 LangGraph 构建单编排器状态机。
- 结构化提取意图、时长、兴趣、同行人、行动能力和临时约束。
- 提供 Knowledge、Route、Profile、Service、Feedback 五类白名单工具。
- 保存 `plan → tool_call → observation → validation → answer` 执行轨迹。
- 对高风险工具设置参数校验和超时，不让模型直接执行任意代码。

**与普通方案的区别**

普通方案将问题和知识片段拼接后交给 LLM；本方案先判断是否需要查询、规划或反馈，再执行工具并校验结果。输出不仅“像答案”，还对应一个可执行任务。

**比赛展示价值**

可在 UI 中实时展示“读取画像—检索知识—规划路线—检查约束”的执行时间线，技术过程可视化强，能直观回答“为什么不是 ChatGPT 套壳”。

**开发难度：中**

现有 FastAPI、LangChain、Qwen 均可复用。只实现单 Agent 和 5 个工具，避免多 Agent 通信复杂度。

### 3.2 创新二：可信、可治理的景区 RAG

**技术实现**

- FAQ 精确匹配处理票价、开放时间等强规则信息。
- BGE-M3 提供 Dense/Sparse 混合召回。
- BGE Reranker 对候选片段精排。
- 文档切片保存来源、标题、页码、版本和有效期元数据。
- 生成前设置相关性阈值；低于阈值时拒答或转人工。
- 回答展示来源，后台维护 200 题黄金测试集并支持回归评测。

**与普通方案的区别**

普通向量库只追求“搜到相似内容”，本方案同时管理知识版本、证据来源、过期时间和失败样本，使 RAG 成为可运营的内容系统。

**比赛展示价值**

现场上传一份新公告，展示知识在 60 秒内生效；再提出知识库外问题，系统明确拒答。正反两个案例比单纯展示正确答案更能证明可信性。

**开发难度：中**

重点是数据和评测工程，不需要训练新模型。开发阶段可使用 ChromaDB，生产说明中保留 Milvus。

### 3.3 创新三：游客状态驱动的情感数字人

**技术实现**

- Paraformer 完成语音识别，CosyVoice 按句流式合成。
- 将意图、用户情绪、游客类型映射到数字人状态机。
- MuseTalk 负责口型，前端叠加 happy、caring、explaining 等表情状态。
- TTS 参数根据状态调整语速和风格。
- GPU 或数字人服务异常时降级为静态形象和音频播放。

**与普通方案的区别**

普通数字人只把任意文本转换为口型视频；本方案的角色、表情、讲解长度和语音风格由游客状态与服务场景共同决定。

**比赛展示价值**

让同一个问题分别以“儿童”“老人”“投诉游客”身份提问，数字人的措辞、语速和表情出现可解释差异，展示效果强且容易理解。

**开发难度：中高**

模型本身无需改造，难点在流式协议、同步和 GPU 环境。应优先保证 1 个角色、3 种状态和稳定降级，再扩展多角色。

### 3.4 创新四：约束感知的动态路线规划

**技术实现**

- 画像包含兴趣、时间、同行人、行动能力和当前状态。
- 景点数据增加停留时间、路径邻接、坡度/无障碍、开放状态等属性。
- 使用可解释评分和约束过滤生成候选路线，必要时可采用简单图搜索。
- 在天气、拥挤或剩余时间变化时，仅重规划未完成部分。
- 每个推荐景点展示“为何推荐”和“满足了哪些约束”。

**与普通方案的区别**

普通推荐根据标签排序；本方案首先满足硬约束，再优化兴趣和体验，且支持游览过程中的动态调整。

**比赛展示价值**

同一位游客从“3 小时少爬坡”切换为“下雨且只剩 1 小时”，路线即时变化，能展示 Agent、工具调用和个性化的组合价值。

**开发难度：中**

不需要训练推荐模型。通过结构化景点数据、规则约束和图搜索即可实现可信 Demo。

### 3.5 创新五：对话驱动的景区运营闭环

**技术实现**

- 记录匿名事件：问题类别、检索命中、工具状态、延迟、情绪和反馈。
- 聚合高频问题、未命中问题、负面主题、景点关注和时段趋势。
- 将高频未命中问题生成“知识补充待办”。
- 知识更新后自动运行相关回归用例，对比准确率变化。
- 为每次会话生成服务质量分：答案正确性、响应速度、任务完成和反馈。

**与普通方案的区别**

普通后台展示预制客流图；本方案的数据来自实际 AI 服务链路，并能反向驱动知识更新和产品优化。

**比赛展示价值**

完成一次未命中问答后，后台计数实时变化；补充知识后重新测试并通过，构成首尾完整的商业闭环。

**开发难度：中**

先使用 PostgreSQL 事件表和定时聚合即可，不必引入 Kafka、Flink 等超出学生项目规模的基础设施。

### 4.2 Agent 状态

建议最小状态结构：

```json
{
  "session_id": "uuid",
  "request": "带老人游玩3小时，尽量少爬坡",
  "intent": "route_planning",
  "profile": {
    "interests": ["历史文化"],
    "companions": ["elderly"],
    "mobility": "low"
  },
  "constraints": {
    "duration_minutes": 180,
    "avoid_steep_path": true
  },
  "plan": [],
  "tool_results": [],
  "evidence": [],
  "confidence": 0.0,
  "response": ""
}
```

### 4.3 工具边界

| 工具 | 输入 | 输出 | 失败策略 |
|---|---|---|---|
| Knowledge Tool | 问题、筛选条件 | 证据片段、来源、分数 | 低分拒答或转人工 |
| Route Tool | 画像、时长、约束 | 有序景点、耗时、理由 | 放宽软约束并明确提示 |
| Profile Tool | 会话和新偏好 | 合并后的匿名画像 | 失败时只使用本轮信息 |
| Service Tool | 设施/开放状态查询 | 结构化服务数据 | 返回“数据待确认” |
| Feedback Tool | 反馈类型和内容 | 工单号/记录状态 | 本地持久化后异步重试 |

### 4.4 架构原则

1. **模型不直接访问数据库**：所有行为必须经过受控工具。
2. **硬规则优先于生成**：票价、开放时间、路线限制从结构化数据读取。
3. **证据不足必须拒答**：拒答是可信能力，不是失败。
4. **所有链路可降级**：LLM、向量库、TTS、数字人异常时仍保留基础服务。
5. **先单 Agent 后多 Agent**：现阶段复杂多 Agent 不会显著增加用户价值。

## 5. 功能模块重设计

### 5.1 游客端

**基础层**

- 文本和语音问答。
- 景点、票务、交通、餐饮、设施查询。
- 按时间和兴趣生成路线。

**比赛增强层**

- Agent 执行状态时间线。
- 回答来源和知识更新时间。
- 知识不足时拒答、反馈或转人工。
- 同行人、行动能力和讲解风格设置。
- “累了、下雨、时间不足”等行中重规划。
- 路线推荐理由与约束满足说明。
- 游客画像查看和一键重置。
- 无障碍模式：大字号、慢语速、简洁讲解。

### 5.2 数字人端

**基础层**

- 语音合成、口型同步和基础表情。
- 形象、音色、语速配置。

**比赛增强层**

- 文本分句后流式 TTS，减少完整答案等待。
- 情绪状态机统一控制语音风格和前端表情。
- 历史讲解员、亲子研学员、服务助手三种角色模板。
- 角色共用同一知识库和工具，避免重复业务逻辑。
- GPU 异常时无感切换到静态头像 + TTS。
- 后台查看数字人生成耗时、失败率和降级次数。

### 5.3 管理后台

**知识库**

- 文档列表、上传、删除和搜索。
- 切片预览、来源、版本、有效期。
- 单题测试和批量回归评测。
- 高频未命中问题转知识待办。

**游客画像**

- 匿名兴趣、同行结构、时长和路线偏好分布。
- 画像维度与满意度、任务完成率交叉分析。
- 明确画像数据保留周期和重置机制。

**服务质量**

- 准确率、Recall@5、忠实度和正确拒答率。
- P50/P95 响应时间、工具成功率、降级率。
- 低分会话列表和失败链路回放。

**运营分析**

- 热点问题、景点关注、负面主题和时段趋势。
- 知识更新前后效果对比。
- 自动生成可执行建议，但由运营人员确认后生效。

## 6. 技术栈取舍

### 6.1 建议保留

| 技术 | 结论 | 原因 |
|---|---|---|
| Vue 3 + TypeScript + Vite | 保留 | 双端原型已形成，工程成熟 |
| FastAPI + WebSocket | 保留 | 适合流式对话和 AI 服务集成 |
| Qwen | 保留 | 中文能力、国产化和兼容 API 适合比赛 |
| BGE-M3 | 保留并落地 | 可支持中文混合检索 |
| ChromaDB / Milvus | 分阶段保留 | ChromaDB 适合开发，Milvus 适合生产表达 |
| Paraformer + CosyVoice | 保留 | 中文语音链路完整，开源可部署 |
| MuseTalk | 保留 | 与现有 2D 数字人路线一致 |
| PostgreSQL + Redis | 保留 | 足够支撑画像、事件、缓存和会话 |

### 6.2 建议升级

| 当前做法 | 升级方案 |
|---|---|
| FAQ/关键词检索 | BGE-M3 混合召回 + BGE Reranker |
| 回答不展示证据 | 来源卡片、相关性分数、知识更新时间 |
| 单条固定对话链 | LangGraph 单 Agent 状态机 |
| 兴趣标签排序 | 硬约束过滤 + 可解释评分 + 简单图搜索 |
| 规则情感分析 | 规则保底 + Qwen/SenseVoice 结构化情绪分类 |
| Mock 运营数据 | 匿名事件表 + 聚合 API |
| 口号式指标 | 固定测试集 + 自动脚本 + 版本对比 |

### 6.3 是否需要 Agent 框架

**需要，但只加入轻量编排框架。**

LangGraph 的价值是：

- 明确每一步状态和失败处理。
- 方便在演示中展示工具调用轨迹。
- 支持重试、超时和分支。
- 与现有 LangChain 依赖和 Python 服务兼容。

不建议在 30 天内构建多个自由协作 Agent。多 Agent 会增加 Token、延迟和调试成本，却很难产生评委可感知的额外价值。

### 6.4 是否需要评测体系

**必须加入。** 没有评测体系，就无法证明“准确率 ≥ 90%”“比普通方案更可信”。

最小评测组件：

1. `datasets/scenic_qa.jsonl`：问题、标准答案、标准证据、类型。
2. RAG 检索评测：Recall@K、MRR。
3. 生成评测：事实正确、引用支持、拒答。
4. Agent 评测：工具选择正确率、参数正确率、任务完成率。
5. 性能评测：文本首字、语音首包、端到端 P95。
6. 版本报告：每次知识或提示词变更前后对比。

## 7. 量化指标设计

### 7.1 指标表

| 分类 | 指标 | 目标 | 样本与方法 |
|---|---|---:|---|
| RAG | 事实问答准确率 | ≥ 90% | 200 道景区问题，人工核验 |
| RAG | Recall@5 | ≥ 90% | 标准证据是否出现在前 5 条 |
| RAG | 回答忠实度 | ≥ 85% | 回答事实是否全部有证据支持 |
| RAG | 正确拒答率 | ≥ 90% | 50 道知识库外问题 |
| Agent | 工具选择正确率 | ≥ 90% | 100 条多意图任务 |
| Agent | 任务完成率 | ≥ 90% | 路线、查询、反馈等任务是否完成 |
| 推荐 | 路线约束满足率 | ≥ 95% | 100 组时长和行动约束 |
| 推荐 | 满意度 | ≥ 4.2/5 | 至少 30 名体验者 |
| ASR | 安静环境字错率 CER | ≤ 10% | 100 条普通话景区问句 |
| ASR | 轻噪声环境 CER | ≤ 18% | 同一语料叠加景区背景音 |
| 性能 | 文本首字响应 P95 | ≤ 1.5 秒 | 100 次请求 |
| 性能 | 语音首句播报 P95 | ≤ 5 秒 | 停止录音到音频开始播放 |
| 性能 | 50 会话并发错误率 | < 1% | 10 分钟持续压测 |
| 知识 | 更新生效时间 | ≤ 60 秒 | 上传文档到可检索 |
| 数字人 | 口型主观自然度 | ≥ 4.0/5 | 30 人盲评 20 段视频 |
| 稳定性 | 核心 Demo 成功率 | ≥ 95% | 连续运行 20 次标准脚本 |

### 7.2 指标表达规范

- 未测试前写“目标”，不能写“已达到”。
- 报告平均值时同时给出 P95，避免掩盖长尾延迟。
- 准确率必须说明题量、来源和判定规则。
- 满意度必须说明人数和量表。
- Demo 数据与真实采集数据在 UI 中明确标记。

## 9. 总体优化优先级

### 高优先级：直接决定比赛可信度

1. 选择一个具体景区，建立可核验知识库和 200 题评测集。
2. 落地混合 RAG、Reranker、来源引用和阈值拒答。
3. 实现单 Agent 编排与 3–5 个白名单工具。
4. 接通语音到数字人播报的最短稳定链路。
5. 运营后台接入真实匿名事件，去除核心图表的纯 Mock 依赖。
6. 固化 5 分钟 Demo 脚本并连续回归。

### 中优先级：形成明显体验差异

1. 游客画像增加同行人、行动能力、讲解偏好。
2. 路线支持硬约束、推荐理由和行中重规划。
3. 增加三种情绪状态和两至三个角色模板。
4. 建立服务质检页、失败链路回放和知识待办。
5. 完成 50 会话压测、30 人满意度测试。

### 低优先级：赛后产品扩展

1. 高精度 3D 数字人和复杂动作捕捉。
2. 多景区跨域推荐和联邦学习。
3. 大规模实时客流预测平台。
4. 多 Agent 自由协同。
5. 自研大模型或大规模微调。

## 10. 30 天实施建议

### 第 1 周：先解决“可信”

- 确定演示景区和数据来源。
- 清洗知识文档并加入来源、版本、有效期。
- 建立 200 题评测集和 50 题拒答集。
- 实现 BGE-M3 召回、Reranker、引用和阈值。
- 输出第一版评测报告。

**退出条件**：知识问答可复现，错误和拒答均有案例。

### 第 2 周：再解决“会做事”

- 引入 LangGraph 单编排器。
- 实现 Knowledge、Route、Profile、Feedback 工具。
- 扩充景点约束字段。
- 游客端展示工具调用轨迹和路线理由。
- 为 Agent 增加 100 条任务测试集。

**退出条件**：复杂路线目标可通过工具完成，不依赖单次自由生成。

### 第 3 周：完成“看得见的多模态”

- 接入 Paraformer 或稳定 ASR 服务。
- 接入 CosyVoice 或稳定流式 TTS。
- 接通 MuseTalk 单角色口型链路。
- 完成三种情绪状态和降级策略。
- 测量端到端 P95 延迟。

**退出条件**：至少一个部署环境可稳定完成语音问答与数字人播报。

### 第 4 周：形成“产品闭环”

- 记录真实匿名事件并生成管理指标。
- 完成知识缺口到回归测试闭环。
- 进行 50 会话压测和 30 人体验测试。
- 录制演示视频、补充 README 截图。
- 连续执行 20 次标准 Demo 并修复失败点。

**退出条件**：评委能在 5 分钟内看到游客价值、技术证据和景区价值。

## 11. 商业化建议

### 11.1 目标客户

- 4A/5A 景区：游客量大、重复咨询多、需要统一服务质量。
- 博物馆和文化场馆：知识深度高，适合角色化讲解。
- 文旅综合体：服务点多，适合路线和消费推荐。
- 研学基地：需要按年龄和课程生成差异化讲解。

### 11.2 收费模式

| 模式 | 适用客户 | 收费构成 |
|---|---|---|
| SaaS 年订阅 | 中小景区 | 景区账号、调用量、知识容量 |
| 私有化部署 | 大型景区/政企 | 软件授权、部署实施、年度维护 |
| 数字人定制 | 品牌景区 | 形象、服装、角色、专属音色 |
| 运营增值 | 有精细化运营需求的客户 | 知识治理、月度洞察、服务优化 |

### 11.3 商业验证指标

- AI 独立完成的咨询占比。
- 人工重复咨询量下降比例。
- 知识更新平均周期。
- 游客任务完成率和满意度。
- 负面问题发现到处理的周期。
- 单次服务推理成本与日均服务量。

### 11.4 风险与应对

| 风险 | 应对 |
|---|---|
| 模型幻觉 | 来源引用、阈值拒答、硬规则数据和人工接管 |
| GPU 成本 | 数字人按需启用，支持静态形象 + TTS 降级 |
| 隐私风险 | 匿名画像、最小采集、可重置、设置保留期限 |
| 景区数据过期 | 知识版本和有效期、后台更新、回归评测 |
| 网络不稳定 | 缓存高频 FAQ、本地小模型或规则降级 |
| Demo 不稳定 | 固定数据快照、健康检查、全链路降级和重复演练 |

## 12. 最终建议

如果只能选择一条升级主线，应选择：

> **可信 RAG + 单 Agent 工具调用 + 数字人稳定输出 + 真实运营闭环。**

这条主线完全保留现有技术路线，不要求训练大模型，也不依赖难以完成的重型平台，却能同时提升创新性、技术深度、功能完整度、行业实用性和商业价值。评委最终应看到的不是更多技术名词，而是一套能够被景区配置、被游客使用、被指标验证并被运营人员持续改进的 AI 产品。
